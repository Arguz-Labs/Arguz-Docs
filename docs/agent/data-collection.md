# Data Collection

This document describes what the Arguz agents capture from your Kubernetes cluster and how that data is represented in the product.

## Categories of data collected

### 1. Deployment metadata

When a Kubernetes Deployment is created, updated, or deleted, the Discovery Agent captures:

- deployment name and namespace
- container images and tags
- desired and available replicas
- rollout strategy
- labels and annotations
- change timestamp
- Helm release metadata when the workload is Helm-managed

### 2. Pod lifecycle signals

- `CrashLoopBackOff`
- `OOMKilled`
- restart counts
- phase transitions such as `Pending`, `Running`, `Succeeded` and `Failed`

### 3. HPA configuration

- API version
- minimum and maximum replicas
- target metrics
- scaling behavior
- current status at capture time

### 4. Namespace and service discovery

- namespace names, labels and annotations
- service names, types, selectors and ports

### 5. Additional resource discovery

- ConfigMaps
- Jobs
- Ingresses
- NetworkPolicies
- RBAC resource creation metadata

### 6. CronJob definitions

- raw cron expression
- readable schedule generated by the product
- timezone
- concurrency policy
- suspend flag
- last schedule
- last successful run

### 7. Job execution history

For each Job created by a tracked CronJob, Arguz stores:

- Job name
- execution status
- start and finish times
- duration
- failure reason
- last 100 log lines for failed runs

### 8. Node resource snapshots

The agent periodically captures:

- node name
- capacity and allocatable CPU, memory, storage and pods
- node conditions such as `Ready`, `MemoryPressure` and `DiskPressure`
- runtime, OS image, architecture, instance type, region and zone when available

!!! note
    Node snapshots are inventory snapshots. Values shown in Arguz for CPU and memory are capacity and allocatable values, not live `kubectl top` usage.

### 9. Cluster metadata heartbeat

At regular intervals, the agent sends:

- Kubernetes version
- cloud provider information
- cluster name, scope, region and zone when available
- agent health status
- node count

### 10. Helm release metadata

If a workload is managed by Helm, Arguz extracts:

- release name
- chart name and version
- Helm revision number
- last deployed timestamp

Sensitive payload content is sanitized before being stored, while resource names remain visible for troubleshooting.

## How data is collected

The agent uses Kubernetes informers plus periodic reconciliation loops:

- watch-based sync for changing resources
- batch submission to reduce API chattiness
- leader election for single-writer behavior

## Configuration options

| Option | Effect |
|---|---|
| `ignore_namespaces` | Exclude namespaces from collection |
| `ignore_resources` | Exclude matching resource globs |
| `ignore_node_labels` | Prevent selected node labels from being sent |

```yaml
log_level: info
ignore_namespaces:
  - kube-system
ignore_resources:
  - job/*
  - cronjob/*
ignore_node_labels:
  - beta.kubernetes.io/arch
```

## What is not stored locally

The agent is stateless. It does not maintain a durable local history database. Inventory is kept in memory and continuously reconciled with the platform.
